# 捣蛋猪，但是粘粘世界

游戏的目的和捣蛋猪（Bad piggies）类似：在起始区域建造载具（或者任何能work的结构），按下开始键后，操作载具到达终点即可通过关卡。如果载具行驶中途出了问题，玩家按中止键即可回到建造载具界面，之前的建造方案会保留。

建造载具：采用粘粘世界（World of goo）的方式。玩家从仓库中拖出节点，节点相距较近时会自动连接，形成较硬的弹簧。

## 实现框架

使用 Unity 3d URP 进行渲染，但是使用 2d 碰撞箱和刚体运算游戏逻辑。

游戏逻辑要抽象出来，不能和渲染逻辑和 UI 逻辑混在一起，最好用一个独立目录。

在实现中，选关页面的 UI 的 game object 预先存在于 scene 中，而关卡内部的 UI 由某个关卡管理器的代码在进入 scene 时生成，这是为了避免和地图冲突。（这意味着关卡内部的 UI 是硬编码的，但是大小、形状、颜色留公共字段。）

需要抽象出一种按钮组件`Button`，以及一种`IButtonReceiver`，有一个方法`public bool OnButtonDown(string buttonName)`。`Button`有公共字段`ButtonName`和`ButtonReceiver`。

## 美术风格

所有的元素用圆形、三角形、长方形来表达。不允许出现其他形状。

不允许出现文字和数字。

## 关卡

一共有许多关卡，可以在选关页面里选择关卡，通关后会自动进入下一关。

选关页面是一个独立的scene，每个关卡都是一个scene。

无论以何种方式离开关卡（在关卡内按左上角的退出按键，或胜利后进入下一关），建造方案都会被存档。

## 选关页面

右下角有一个正方形按钮，按了就会退出游戏。

中间有 `TotalLevelNum = 10` 个正方形按钮。如果关卡已经通过，则为绿色；如果解锁了但未通过，则为蓝色；如果未解锁，则为灰色。

点击已经解锁的关卡的按钮可以进入关卡。

初始解锁前 `InitialUnlockedLevelNum = 3` 个关卡。每通过一个关卡，就按顺序解锁一个关卡。

存档文件中，只需要保存通过的关卡列表 `CompletedLevelList`，因为解锁的关卡总是前 `CompletedLevelList.Length + InitialUnlockedLevelNum` 个关卡。

`CompletedLevelList` 在通过关卡时会立即保存。

## 关卡内部

### 物理

地面：一些不移动的，有碰撞箱的刚体，很可能是多边形。

终点：一个不移动的，有碰撞箱的 `IsTrigger` 的多边形。

节点：载具的组成部分，有碰撞箱的刚体。某些特定种类的节点可能不碰撞。

（注意，节点之间的弹簧链接是一种力的关系，并非刚体，没有质量和碰撞箱。）

（地面和终点每关不一样，所以显然保存在scene中。）

### 界面

在任何时刻，左上角都有一个向左的三角形按钮，点击可以退出到选关页面。

界面有建造、运行、过关三种状态。

#### 建造

地图上有一个固定的建造区域，并有一个固定的建造时的摄像头位置，这些都保存在scene中。

建造时，屏幕下方有一个仓库区域，里面有一些节点。仓库和里面的节点并不真的在物理上存在，只是贴图，节点会只有在开始运行时才会在物理上生效。

可以用鼠标拖动节点。可以在仓库和建造区域这两个区域里自由地拖到任何地方。如果拖到了不合法的位置（比如这两个区域之外，或者与其它节点重合了），那么这次拖动就失败，节点会回到原来的位置。

节点拖动到建造区域中后，如果周围有其它节点，就会连接起来。具体见机制：节点。

屏幕右下角有一个圆形按钮，点击就可以开始运行。（建造蓝图会被存档，既在内存上存也在硬盘上存。）

#### 运行

在运行界面，开始运行的按钮会被中止按钮代替，这是一个正方形。按下它就回到建造模式，读取内存上的蓝图。（当然，同时要销毁所有地图中的物理节点。）

在开始运行时，节点会获得物理实体，并开始在地图中运动。相机会以特定的逻辑跟随节点，具体见机制：运行时相机。

运行时，点击特定的节点可能会产生特定效果，因此在代码上留一个运行时节点被点击的接口。

如果任何一个节点碰到了终点，那么进入过关状态。

#### 过关

过关界面和运行界面非常接近。节点的物理演算继续进行，相机的逻辑不变。唯一区别在于右下角的中止按钮被一个向右的三角形代替，按下可以进入下一关。如果是最后一关，那么按下后退回到选关页面。

进入过关状态时，更新`CompletedLevelList` 并立即保存。

## 机制

### 节点

节点可能会与其它节点连接。在建造阶段，节点决定了自己与哪些节点连接；在运行阶段，节点的连接产生物理特性。

节点有不同类型。继承自同一个类`Node`。

节点的数量不会特别多（估计不会超过20），所以使用O(n^2)算法没有任何问题。

### 节点如何决定连接

这一部分逻辑可能需要后续调整，所以需要抽象出来。

节点可以主动与其它节点连接，也可以被动地被其他节点连接。如果两个节点互相主动连接，也只会产生一个连接，不会重复产生两个。

节点的主动连接有一个最大半径字段`MaxConnectRadius`，和一个最大连接数字段`MaxConnectNumber = 3`。被动连接没有此限制。

节点被拖动时，其它节点对它的主动连接会保留，但如果这个连接的长度超过了对方的`MaxConnectRadius`，或是节点被放进了仓库，那么这个连接会断掉。

节点被放下时（拖动就位时；拖动失败的回归原位不算），找到`MaxConnectRadius`内最近的`MaxConnectNumber`个没有主动连接自己的节点，对其进行主动连接。（主动连接一旦确定，只要自己没有被拖动，其它节点的移动就不会更新这个节点的主动连接。）

节点被拖动时会显示将会形成的连接的预览。

### 节点和连接的物理特性

所有节点都有质量。

有些节点和其他东西物理碰撞，有些节点则不碰撞。不碰撞的节点只通过连接和其他节点交换力。

连接不碰撞。

连接在径向上的力作用是弹簧。弹簧的原长是开始运行时节点之间的距离。节点的弹性系数`SpringK`暂时使用节点类的公共静态字段。（还要加一个正比于速度的衰减力，避免弹簧抽搐，参数也用公共静态字段。）

弹簧上的长度变化量的绝对值超过公共静态字段`SpringBreakLength`后，连接就断裂消失了。

连接不止可以转递径向力，也有可能传递切向力。传递的切向力大小由两端受到的力矩决定。满足角动量守恒，即切向力大小等于力矩除以连接长度。（这当然意味着如果两端都是光滑铰接，那么传递的切向力始终是零。）

有些节点不会旋转，有些节点会旋转。（不碰撞的节点一定不会旋转。）（不会旋转的节点一定不会与连接有力矩交换，且要么不碰撞，要么碰撞箱是完全光滑的圆。）

会旋转的节点的方向与连接的方向是耦合的，可能会因此产生力矩。机制如下：

会旋转的节点在开始运行时记录所有连接的初始角度。每个会旋转的节点还会记录所有连接上次`FixedUpdate`的角度。

注意：连接的角度是一个特殊的东西。角度是一个浮点数，并且没有上下限。如果连接转了一圈，那么应当认为角度增加了2pi，而不是不变。这意味着连接的角度和历史有关，并不能通过当前的相对位置直接得到，故需要用特殊的方式来记录。

每一个`FixedUpdate`，会旋转的节点计算每个连接的角度。先用x y相对位置的arctan来计算出没有历史的角度，然后根据上一帧的角度，认为这个角度在一帧内不会改变超过pi，得到这一帧的角度。

接着，节点与连接产生力矩交互，力矩大小正比与「连接相对于物品的角度」这个值与初始值的差值，类似于一个角向弹簧。类似径向力，参数使用公共字段，而且也有衰减力矩。

### 运行时相机

相机的中心位置（x和y）跟随所有节点的物理质心。

相机默认的缩放（z）与建造时的固定值一致。但是如果某个节点里摄像头中心太远，摄像头就会实时拉远，使得节点到中心的水平距离不会超过`MaxHorizontalSpan = 0.4`个屏幕，竖直距离不会超过`MaxVerticalSpan = 0.4`个屏幕。

相机的xyz有一定指数平滑，特征时间留公共字段。
